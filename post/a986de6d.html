DDIA：数据模型与查询语言

bigboss2063
2023-05-01
阅读笔记
DDIA

数据模型可能是软件开发中最重要的部分了，因为它们的影响如此深远：不仅仅影响着软件的编写方式，而且影响着我们的 解题思路。

多数应用使用层层叠加的数据模型构建。对于每层数据模型的关键问题是：它是如何用低一层数据模型来 表示 的？

多层应用的基本思想就是每一层都以一个明确的数据模型来隐藏更低层次中的复杂性，使得工作在不同层次的人可以互相合作。

关系模型与文档模型

现在最著名的数据模型是 SQL，它基于关系模型，关系模型将事务处理、批处理等实现细节隐藏在更简洁的接口之后。

关系模型的每个竞争者都在其时代产生了大量的炒作，但从来没有持续

NoSQL 的诞生

NoSQL 被追溯性地重新解释为 不仅是 SQL（Not Only SQL）。

NoSQL 数据库的驱动因素：

  • 为了存储更大的数据集和获得更大的吞吐量，需要比关系型型数据更好的伸缩性；
  • 免费和开源软件更受偏爱；
  • 关系模型不能很好地支持一些特殊的查询操作；
  • 受挫于关系模型的限制性，渴望一种更具多动态性与表现力的数据模型。

考虑到现实中并没有“银弹”，所以在可预见的未来仍然需要 SQL 和 NoSQL 合作。

对象关系不匹配

将面向对象语言中的类转化为关系型数据库中的表、行、列时，需要一个转换层，会使用 ORM 框架来完成这个操作。

而文档模型在这方面有一定的优势。JSON 表示相比于 SQL 的多表模式有更好的局部性，不需要进行多表联查，相关信息都在一个地方，一次查询就足够了。一对多关系会构成一个树形结构，而使用 JSON 来表示这个树形结构会更加明确（不用从上层开始查询之后，再生成孩子节点，而是一整颗树在一个 JSON 中）。

多对一和多对多关系

在多对一和多对多关系中，是存储引用（Record ID）还是存储对人类有意义的信息（文本字符串）呢？

使用 ID 的好处是，ID 对人类没有任何意义，因而永远不需要改变：ID 可以保持不变，即使它标识的信息发生变化。任何对人类有意义的东西都可能需要在将来某个时候改变 —— 如果这些信息被复制，所有的冗余副本都需要更新。这会导致写入开销，也存在不一致的风险（一些副本被更新了，还有些副本没有被更新）。去除此类重复是数据库 规范化（normalization） 的关键思想。

我在这里的理解是，如果使用的数据库是支持连接的，那么可以存储引用，如果对连接支持不好的数据库要进行连接操作就只能在应用程序中进行。这样存储文本字符串会好一点，不需要连接，直接得到结果。

但是这也不能够确定，因为即便应用程序的最初版本适合无连接的文档模型，随着功能添加到应用程序中，数据会变得更加互联。

文档数据库是否在重蹈覆辙？

作者这里的意思是文档模型是否又回到了网络模型。

关系模型和网络模型的提出是用于解决层次模型的局限性的。IMS 能良好处理一对多的关系，但是很难应对多对多的关系，并且不支持连接。

网络模型每条记录可能有多个父节点，这样就允许多对一和多对多建模。

网状模型中记录之间的链接不是外键，而更像编程语言中的指针（同时仍然存储在磁盘上）。访问记录的唯一方法是跟随从根记录起沿这些链路所形成的路径。这被称为 访问路径（access path）。

最简单的情况下，访问路径类似遍历链表：从列表头开始，每次查看一条记录，直到找到所需的记录。但在多对多关系的情况中，数条不同的路径可以到达相同的记录，网状模型的程序员必须跟踪这些不同的访问路径。

后面的内容有一点没看明白，但是感觉大致就是用网络模型存储数据进行查询时，必须要手写大量的数据库查询代码，如果模型发生变化，这里写的所有代码都需要发生变化，非常不灵活。

相比之下，在关系型数据库中，程序员只要指明自己想要哪些符合条件的数据，剩下的都交给数据库来决定，相当于对程序员是透明的（CMU 15-445 中提到过这一点，应该是一个意思）。由查询优化器来决定按照什么顺序执行，使用哪些索引等。

而文档模型，在一对多关系下，就是层次模型，而在多对一和多对多关系中，引用的项目通过一个唯一标识符文档引用指出，跟关系型数据的外键没有本质区别。

所以文档数据库并没有重蹈覆辙。

关系型数据库与文档数据库在今日的对比

支持文档数据模型的主要论据是架构灵活性，因局部性而拥有更好的性能，以及对于某些应用程序而言更接近于应用程序使用的数据结构。关系模型通过为连接提供更好的支持以及支持多对一和多对多的关系来反击。

哪种数据模型更有助于简化应用代码？

当满足以下条件时选择文档模型：

  • 应用程序中的数据类似于文档结构（即，一对多关系树，通常一次性加载整个树）；
  • 文档嵌套不要太深；
  • 不会用到多对一或多对多关系。

否则还是选择关系模型。

文档模式中的 schema 灵活性

ps：这个 schema 感觉翻译成模式怪怪的，还是就是叫 schema 吧。

文档数据库不会强制要求文档中的数据采取某种 schema（我的理解是不会像关系型数据库一样，需要按照建表时的字段、约束来对数据进行操作）。这被称作 schemaless。

但更准确的来说是应该是 schema-on-read，因为数据读取的时候，应用程序总是会按照某种 schema 来进行检查。相应的，关系型数据库就是 schema-on-write。类似于静态/动态类型检查。

在应用程序想要改变其数据格式的情况下，这些方法之间的区别尤其明显。

例如，假设你把每个用户的全名存储在一个字段中，而现在想分别存储名字和姓氏【23】。在文档数据库中，只需开始写入具有新字段的新文档，并在应用程序中使用代码来处理读取旧文档的情况。

另一方面，在 “静态类型” 数据库模式中，通常会执行以下 迁移（migration） 操作。

所以当集合中的项目不都是相同的结构时（schema 不同），使用文档模式是一种更优解。反之，关系模型是一种更好的选择。

查询的数据局部性

文档通常以单个连续字符串形式进行存储。所以当需要一整个文档时，存储局部性会带来优势，因为不需要去多个表中依次查询。

但这存在局限性，文档数据库通常只能一次加载整个文档，即使只需要一小部分，同样，更新也只能重写整个文档。当文档很大的时候，对性能影响就大了。为了避免这个问题又只能保持相对较小的文档，而这样又限制了文档数据库的适用场景。

文档和关系数据库的融合

关系型数据库开始支持 XML 和 JSON 的查询存储，而文档数据库也开始支持连接。

随着时间的推移，关系数据库和文档数据库似乎变得越来越相似，这是一件好事：数据模型相互补充 ，如果一个数据库能够处理类似文档的数据，并能够对其执行关系查询，那么应用程序就可以使用最符合其需求的功能组合。

关系模型和文档模型的混合是未来数据库一条很好的路线。

数据查询语言

当引入关系模型时，关系模型包含了一种查询数据的新方法：SQL 是一种 声明式 查询语言，而 IMS 和 CODASYL 使用 命令式 代码来查询数据库。

命令式语言告诉计算机以特定顺序执行某些操作。

在声明式查询语言（如 SQL 或关系代数）中，你只需指定所需数据的 schema，结果必须符合哪些条件，以及如何将数据转换（例如，排序，分组和集合），但不是如何实现这一目标。数据库系统的查询优化器决定使用哪些索引和哪些连接方法，以及以何种顺序执行查询的各个部分。

SQL 并不依赖于排序，所以有着比命令式语言更高的并行性。由于命令式语言代码指定了特定的顺序，那么无法很好发挥多个 CPU 的性能。

NoSQL 系统可能会意外发现自己只是重新发明了一套经过乔装改扮的 SQL。

图数据模型

关系模型可以处理多对多关系的简单情况，但是随着数据之间的连接变得更加复杂，将数据建模为图形显得更加自然。

属性图

在关系型数据库中的图存储可以看作由两个关系组成，一个表存储顶点，一个表存储边。它的优势是：

  1. 任何顶点都可以有一条边连接到任何其他顶点。没有模式限制哪种事物可不可以关联。
  2. 给定任何顶点，可以高效地找到它的入边和出边，从而遍历图，即沿着一系列顶点的路径前后移动。
  3. 通过对不同类型的关系使用不同的标签，可以在一个图中存储几种不同的信息，同时仍然保持一个清晰的数据模型。

SQL 中的图查询

相比于使用图数据库中的查询语句，使用 SQL 进行图查询非常笨拙，可能需要十分复杂的步骤。我们应该根据应用程序选择数据模型，强扭的瓜不甜。

三元组存储和 SPARQL

三元组存储模式大体上与属性图模型相同，用不同的词来描述相同的想法。

在三元组存储中，所有信息都以非常简单的三部分表示形式存储（主语，谓语，宾语）。

三元组的主语是图中的一个顶点，宾语则可能是原始数据类型的值或者图中的另一个顶点。

语义网

（基于三元组的）资源描述框架（RDF）【41】，被用作不同网站以统一的格式发布数据的一种机制，允许来自不同网站的数据自动合并成 一个数据网络 —— 成为一种互联网范围内的 “通用语义网数据库”。

不幸的是，语义网在二十一世纪初被过度炒作，但到目前为止没有任何迹象表明已在实践中应用，这使得许多人嗤之以鼻。它还饱受眼花缭乱的缩略词、过于复杂的标准提案和狂妄自大的困扰。（该书出版的时间应该是 2018 年）

总结

关系模型、文档模型、图模型在今天都被广泛使用，并且在各自的领域都发挥很好。

  • 关系模型：适用于多对多和多对一关系；
  • 文档模型：主要关注自我包含的数据文档（局部性），而且文档之间的关系非常稀少；
  • 图模型：任意事物之间都可能存在潜在的关联。

虽然一个模型可以模拟另一个模型，但往往效果不好并且难度较大。所以针对不同的目的使用不同的数据模型是非常重要的。
